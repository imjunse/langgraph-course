Overview

LangGraph란?

고도화된 AI 에이전트를 구축, 관리, 배포하기 위한 저수준(Low-Level) 오케스트레이션 프레임워크이자 런타임. 개발자가 에이전트의 사고 과정과 행동 순서를 한 땀 한 땀 설계할 수 있는 투명한 설계도를 제공합니다.

"기존 LangChain 에이전트가 내부 로직이 꽁꽁 숨겨진 블랙박스였다면,
LangGraph는 에이전트가 어떤 고민을 하고 어떤 순서로 움직일지
개발자가 가이드할 수 있는 투명한 설계도를 제공합니다."
-- 강의 중 발췌

Core Concepts

4가지 핵심 개념

LangGraph의 모든 에이전트는 이 네 가지 빌딩 블록으로 구성됩니다. 강의 전체에서 반복적으로 등장하는 가장 중요한 개념입니다.

📖

State

스테이트 = 공유 공책

노드들이 작업 결과를 기록하는 공유 데이터 구조. Python TypedDict 클래스로 정의하며, 축구공처럼 노드 사이를 전달됩니다. messages 필드에 add_messages 함수를 지정하면 메시지가 자동으로 append됩니다.

⚙

Node

노드 = 작업자 (함수)

데이터를 처리하는 작업자. Python def 함수로 구현하며, 반드시 State를 입력 파라미터로 받아 작업 후 결과를 State 형태의 dict로 반환합니다. LLM Call 노드, Tool 노드 등이 있습니다.

➔

Edge

엣지 = 연결선 (실선)

노드와 노드를 연결하는 실선. add_edge(START, "llm")처럼 작업 흐름의 순서를 명시적으로 정의합니다. 항상 고정된 경로로 이동합니다.

🔀

Conditional Edge

컨디셔널 엣지 = 라우터 (점선)

LLM의 추론 결과에 따라 다음 노드를 동적으로 결정. tool_calls가 있으면 Tool 노드로, 없으면 END로 라우팅하는 조건 분기입니다.

🎓 강의 개요

LangGraph란?

고도화된 에이전트를 구축, 관리, 배포하기 위한 저수준(Low-Level) 오케스트레이션 프레임워크이자 런타임. LangChain이 블랙박스였다면, LangGraph는 에이전트의 사고 과정을 개발자가 한 땀 한 땀 가이드할 수 있는 투명한 설계도.

💡 왜 LangChain도 설치하나?
LangGraph에서 에이전트를 만들 때 LLM이 필요한데, 언어 모델 객체를 직접 구현할 필요 없이 LangChain에 이미 통합된 패키지(ChatGoogleGenerativeAI 등)를 그대로 가져다 쓸 수 있기 때문. 더 로우레벨까지 직접 구현하고 싶다면 LangChain 없이도 가능.

1️⃣ Step 1: 모델 & 툴 정의

모델 생성

LangChain에 통합된 ChatGoogleGenerativeAI 객체를 사용. 모델명과 하이퍼파라미터(temperature 등)를 설정해 LLM 객체 생성.

툴 정의 & 바인딩

LangChain의 @tool 데코레이터로 도구 함수 작성 (사칙연산 등). model.bind_tools(tools)로 모델과 도구를 하나의 바인딩된 새 객체로 결합.

from langchain_google_genai import ChatGoogleGenerativeAI from langchain_core.tools import tool model = ChatGoogleGenerativeAI(model="gemini-3-flash") # @tool 데코레이터로 도구 정의 @tool def multiply(a: int, b: int) -> int: """두 수를 곱합니다.""" return a * b tools = [multiply, add, divide] model_with_tools = model.bind_tools(tools) # 바인딩!

2️⃣ Step 2: State 정의 (공책 설계)

State = 노드 간 공유되는 데이터 구조

TypedDict 클래스로 정의. messages 필드에 Annotated[list[AnyMessage], add_messages]를 사용하면 새 메시지가 들어올 때 자동으로 리스트 뒤에 append됩니다. add_messages는 LangGraph 개발자들이 미리 만들어 놓은 편의 함수.

from langgraph.graph import StateGraph, START, END from langgraph.graph.message import add_messages from typing import TypedDict, Annotated from langchain_core.messages import AnyMessage class AgentState(TypedDict): messages: Annotated[list[AnyMessage], add_messages] # 자동 append llm_calls: int # LLM 호출 횟수 추적
⚽ 축구 비유
축구 선수(Node)가 공(State)을 패스하듯, 각 노드가 작업한 결과물을 다음 노드에게 State를 통해 전달합니다. 패스할 때 공의 구조(데이터 스키마)가 있어야 하듯, State를 클래스로 정의합니다.

3️⃣ Step 3: Node 정의 (작업자 함수)

LLM Call 노드
  • State의 messages를 읽어 LLM에 invoke
  • 바인딩된 model_with_tools 사용
  • 응답을 messages에 추가 + llm_calls 카운트
  • 시스템 메시지도 여기서 삽입 가능
Tool 노드
  • 마지막 AI 메시지의 tool_calls 확인
  • tools_by_name 딕셔너리로 매칭
  • tool.invoke(arguments)로 실행
  • 결과를 ToolMessage에 담아 반환
  • tool_call_id 필수 (병렬 호출 추적)
# LLM Call 노드 def llm_call(state: AgentState): response = model_with_tools.invoke(state["messages"]) return { "messages": [response], "llm_calls": state.get("llm_calls", 0) + 1 } # Tool 노드 tools_by_name = {t.name: t for t in tools} def tool_node(state: AgentState): results = [] for tc in state["messages"][-1].tool_calls: tool = tools_by_name[tc["name"]] result = tool.invoke(tc["args"]) results.append(ToolMessage( content=str(result), tool_call_id=tc["id"] # 병렬 호출 식별용 )) return {"messages": results}

4️⃣ Step 4-5: Graph 생성, 연결 & 실행

Conditional Edge의 라우팅 로직

LLM 응답에 tool_calls가 있으면 → Tool 노드로 라우팅. 없으면(직접 답변 가능) → END로 라우팅. 이 조건 분기가 Conditional Edge의 핵심.

# 라우팅 함수 def should_continue(state: AgentState): last = state["messages"][-1] if last.tool_calls: return "tools" # tool_calls 있으면 Tool 노드로 return "end" # 없으면 END로 # Graph 생성 & 연결 graph = StateGraph(AgentState) graph.add_node("llm", llm_call) graph.add_node("tools", tool_node) graph.add_edge(START, "llm") # START → LLM (실선) graph.add_conditional_edges("llm", should_continue, {"tools": "tools", "end": END}) # LLM → 조건 분기 (점선) graph.add_edge("tools", "llm") # Tool → LLM (실선, 루프) app = graph.compile() result = app.invoke({"messages": ["3과 23을 곱해줘"]})

💬 메시지 흐름 상세

State.messages 리스트가 채워지는 과정
  • HumanMessage — "3과 23을 곱해줘" (사용자 입력)
  • AIMessage — content="" + tool_calls=[{name:"multiply", args:{a:3,b:23}}]
  • ToolMessage — content="69" + tool_call_id="..." (도구 실행 결과)
  • AIMessage — content="3과 23을 곱한 결과는 69입니다." (최종 답변)
💡 tool_call_id가 필요한 이유
"서울과 도쿄의 날씨를 알려줘"처럼 여러 도구가 병렬 호출될 수 있습니다. 각 ToolMessage가 어떤 tool_call에 대한 응답인지 매칭하기 위해 고유 ID가 필요합니다.

🎵 Orchestration & Worker 패턴

지휘자(Orchestrator) 에이전트

메인 에이전트가 지휘봉을 통해 각기 다른 전문성을 가진 Worker 에이전트들을 하나로 묶고, 이들이 서로 협력하며 최적의 결과물을 만들어 내도록 설계합니다.

Dynamic Worker

LLM의 강력한 추론 능력을 이용해 사용자 요청의 성격/양에 따라 동적으로 에이전트 팀을 꾸리고, 업무를 분배한 뒤 결과를 하나로 취합해 반환하는 다이나믹 구조.

🎵 오케스트라 비유
에이전트라는 개별 연주자들이 제멋대로 연주하지 않도록 지휘봉을 휘두르는 지휘자. State라는 공유 메모리를 기반으로 "지금 검색이 필요한지, 코드를 실행해야 할지, 결과가 부족해 다시 돌아가야 할지" 매 순간 판단합니다.

📈 Evaluator & Optimizer 구조

Evaluator (평가자)

에이전트의 출력 결과를 평가하는 노드. 품질 기준에 따라 결과가 충분한지 판단하고, 부족하면 다시 작업을 지시합니다.

Optimizer (최적화자)

Evaluator의 피드백을 받아 결과를 개선하는 노드. Evaluator ↔ Optimizer 간 반복 루프를 통해 점진적으로 품질을 높입니다.

🛠 2강에서 만든 에이전트를 발전시키는 법

기초 → 고급 아키텍처 확장
  • 단일 LLM + Tool 구조에서 다중 Worker 구조로 확장
  • Conditional Edge를 복합 라우팅으로 발전
  • Evaluator 노드 추가로 품질 보증 루프 구현
  • LangGraph 공식 문서의 아키텍처 패턴을 코드로 실습

💾 메모리 시스템

Checkpoint 기반 메모리

LangGraph는 그래프 실행의 각 단계를 Checkpoint로 자동 저장합니다. 이를 통해 에이전트가 이전 대화를 기억하고, 문맥을 유지한 채 응답할 수 있습니다.

thread_id로 세션 관리

각 대화 세션을 thread_id로 구분. 같은 thread_id로 invoke하면 이전 대화 이력이 State에 자동 로드되어 연속적인 대화가 가능합니다.

⏰ Time Travel (시간 여행)

과거 상태로 되돌아가기

Checkpoint 히스토리를 탐색해 과거의 특정 시점으로 되돌아갈 수 있습니다. 에이전트가 잘못된 판단을 했을 때, 그 시점 이전으로 롤백하여 다른 경로로 재실행할 수 있는 강력한 디버깅 & 제어 기능.

🔀 Forking (분기)

상태를 분기하여 병렬 실험

특정 Checkpoint에서 상태를 복제(Fork)하여 서로 다른 경로를 병렬로 탐색할 수 있습니다. "만약 여기서 다른 도구를 호출했다면?" 같은 What-if 시나리오를 실제로 실행해볼 수 있는 기능.

🎮 게임 세이브 비유
Time Travel = 게임에서 이전 세이브 포인트를 불러오는 것. Forking = 세이브 포인트에서 여러 갈래의 선택지를 각각 플레이해보는 것. Checkpoint가 자동 세이브 역할을 합니다.

🤝 멀티 에이전트 협업

지휘자(Conductor) 에이전트

사용자 요청의 성격과 양에 따라 동적으로 에이전트 팀을 구성합니다. 각 전문 에이전트에게 업무를 분배하고, 결과를 하나로 취합하여 최종 응답을 생성합니다.

전문 에이전트 팀

검색 전문, 코드 실행 전문, 분석 전문 등 각기 다른 전문성을 가진 에이전트들이 하나의 그래프 안에서 협력합니다. 단 하나의 에이전트가 아닌, 여러 에이전트가 서로 협업해 업무를 처리.

✋ Dynamic Interrupt (동적 인터럽트)

Human-in-the-Loop 안전장치

에이전트가 업무 중 막히는 부분이 있으면 실행을 멈추고 사람의 승인이나 수정을 입력받아 동적으로 반영하는 기능. 위험한 작업 전 확인 요청, 불확실한 결과에 대한 검증 등 보다 안전한 에이전트 시스템을 구축할 수 있습니다.

🚀 이 챕터에서 구현하는 것

  • 여러 에이전트가 서로 협업해 업무를 처리하는 서비스
  • 지휘자 에이전트의 동적 결정에 따른 팀 구성 & 업무 분배
  • Dynamic Interrupt로 사람의 승인/수정을 중간에 반영
  • 3강, 4강에서 배운 모든 패턴의 종합 실습
🏢 회사 비유
CEO(지휘자 에이전트)가 프로젝트 성격에 맞는 팀원(Worker 에이전트)을 동적으로 배치하고, 중요한 의사결정은 이사회(Human)의 승인을 받아 진행하는 구조입니다.
How to Build

에이전트 구축 5단계 요약

LangGraph에서 에이전트를 구축할 때 거치는 5가지 단계입니다.

모델 & 툴 정의

ChatGoogleGenerativeAI로 LLM 생성, @tool로 도구 정의, bind_tools()로 바인딩

State 정의

TypedDict 클래스로 messages(Annotated + add_messages), llm_calls 등 필드 설계

Node 정의

llm_call(), tool_node() 함수 작성. 반드시 State를 입력받아 State 형태로 반환

Graph 생성 & 연결

StateGraph + add_node + add_edge(실선) + add_conditional_edges(점선)

컴파일 & 실행

graph.compile() 후 app.invoke()로 실행. thread_id로 메모리 연결

Comparison

LangChain vs LangGraph

LangChain에서 LangGraph로 넘어가야 하는 이유

LangChain

High-Level Agent Framework
  • 내부 로직이 숨겨진 블랙박스
  • create_agent()로 간단히 생성
  • 간단한 Q&A 에이전트에 적합
  • 일회성 구조, 메모리 제한적
  • 빠른 프로토타이핑 용이

LangGraph

Low-Level Orchestration Framework
  • 투명한 설계도 -- 완벽한 통제 & 예측
  • State/Node/Edge를 직접 설계
  • 복잡한 비즈니스 로직 완벽 소화
  • 멀티에이전트 오케스트레이션
  • Time Travel, Forking, Dynamic Interrupt
Getting Started

환경 설정

강의를 따라가기 위한 필수 패키지 설치와 API 키 설정 방법입니다.

📦 LangGraph

pip install langgraph
v1.0.6+ (2025.10 버전 업데이트)

⛓ LangChain

pip install langchain
언어 모델 통합을 위해 함께 설치

🤖 Google Gemini

pip install -U langchain-google-genai
Gen AI (개인/학습용) -- Enterprise는 Vertex AI

🔑 API Key 설정

os.environ["GOOGLE_API_KEY"] = "your-key"
Google AI Studio (aistudio.google.com)에서 발급

🎓 실습 자료

YouTube 고정 댓글 → Google Drive 링크
Colab 파일 → 파일 → 드라이브에 사본 저장

🧠 지원 모델

gemini-3-flash / gemini-3-pro-preview
GPT-5 Nano, Gemini 2.5 Flash 등도 지원
🎵 "오케스트레이션 = 지휘자"
에이전트라는 개별 연주자들이 제멋대로 연주하지 않도록
지휘봉을 휘두르는 지휘자 에이전트가 바로 LangGraph입니다.
State라는 공유 메모리를 기반으로 매 순간 판단하며,
각기 다른 전문성을 가진 에이전트들을 하나로 묶어
최적의 결과물을 만들어냅니다.
